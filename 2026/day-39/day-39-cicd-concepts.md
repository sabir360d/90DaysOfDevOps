# Day 39 – What is CI/CD?

## Task
Before writing a single pipeline, understand why CI/CD exists and what it actually does.

## Task 1: The Problem

Think about a team of 5 developers all pushing code to the same repo manually deploying to production.

1. What can go wrong?

- Code conflicts when multiple developers push changes
- Bugs introduced without proper testing
- Broken production due to manual mistakes
- Deployment inconsistencies across environments
- Lack of rollback strategy

2.  What does "It works on my machine" mean and why is it a real problem?

This means code runs correctly in a developer’s local environment but fails elsewhere due to differences in dependencies, configurations, or environments. It is a real problem because production environments rarely match local setups exactly.

3. How many times a day can a team safely deploy manually?

- Usually 1–2 times per day at most
- High risk limits frequent releases
- Requires coordination and downtime planning

## Task 2: CI vs CD

### Continuous Integration (CI)
- Developers frequently merge code into a shared repository
- Automated builds and tests run on every change
- Helps catch bugs early and ensures code compatibility

**Example:**
A developer pushes code, and tests automatically run using GitHub Actions to verify nothing is broken.

---

### Continuous Delivery (CD)
- Extends CI by preparing code for release
- Code is always in a deployable state
- Deployment to production requires manual approval

**Example:**
After passing tests, the app is automatically deployed to a staging environment, ready for release.

---

### Continuous Deployment (CD)
- Fully automated release process
- Every successful change is deployed to production automatically
- No manual approval required

**Example:**
A SaaS app automatically deploys new features to users after passing all tests.

## Task 3: Pipeline Anatomy

### Trigger
Event that starts the pipeline (e.g., push, pull request)

### Stage
A logical group of jobs (e.g., build, test, deploy)

### Job
A unit of work executed within a stage

### Step
A single command or action inside a job

### Runner
The machine or environment that executes jobs

### Artifact
Files generated by jobs (e.g., binaries, Docker images)

## Task 5: Explore in the Wild

### Repository Chosen
FastAPI (example)

### Workflow File
https://github.com/fastapi/full-stack-fastapi-template/blob/master/.github/workflows/deploy-staging.yml

```yaml
name: Deploy to Staging

on:
  push:
    branches:
      - master

jobs:
  deploy:
    # Do not deploy in the main repository, only in user projects
    if: github.repository_owner != 'fastapi'
    runs-on:
      - self-hosted
      - staging
    env:
      ENVIRONMENT: staging
      DOMAIN: ${{ secrets.DOMAIN_STAGING }}
      STACK_NAME: ${{ secrets.STACK_NAME_STAGING }}
      SECRET_KEY: ${{ secrets.SECRET_KEY }}
      FIRST_SUPERUSER: ${{ secrets.FIRST_SUPERUSER }}
      FIRST_SUPERUSER_PASSWORD: ${{ secrets.FIRST_SUPERUSER_PASSWORD }}
      SMTP_HOST: ${{ secrets.SMTP_HOST }}
      SMTP_USER: ${{ secrets.SMTP_USER }}
      SMTP_PASSWORD: ${{ secrets.SMTP_PASSWORD }}
      EMAILS_FROM_EMAIL: ${{ secrets.EMAILS_FROM_EMAIL }}
      POSTGRES_PASSWORD: ${{ secrets.POSTGRES_PASSWORD }}
      SENTRY_DSN: ${{ secrets.SENTRY_DSN }}
    steps:
      - name: Checkout
        uses: actions/checkout@v6
      - run: docker compose -f compose.yml --project-name ${{ secrets.STACK_NAME_STAGING }} build
      - run: docker compose -f compose.yml --project-name ${{ secrets.STACK_NAME_STAGING }} up -d
```

### Findings

### Trigger
- Runs on push events to the master branch
- This means every time code is pushed to master, deployment is triggered automatically

### Jobs
- Contains 1 job named "deploy"

### Conditions
- The job runs only if the repository owner is NOT 'fastapi'
- This prevents deployments in the official FastAPI repo and only allows it in user forks/projects

### Runner
- Uses a self-hosted runner
- Tagged with "staging", meaning it likely runs on a custom server configured for staging deployments

### Environment Variables
- Uses multiple secrets such as:
  - DOMAIN
  - STACK_NAME
  - SECRET_KEY
  - Database and email credentials
- These are securely stored in GitHub Secrets and injected during runtime

### Steps

1. Checkout Code
   - Uses actions/checkout to pull the latest code from the repository

2. Build Application
   - Runs docker compose build using a staging-specific project name
   - Builds all services defined in compose.yml

3. Deploy Application
   - Runs docker compose up -d
   - Starts containers in detached mode on the staging server

### What does it do? (best guess)

- This workflow is a Continuous Deployment pipeline for staging
- It automatically deploys the application whenever code is pushed to master
- Uses Docker Compose to build and run services
- Runs on a self-hosted staging server instead of GitHub-hosted runners
- Relies heavily on secrets for secure configuration
- This is a real-world example of automated staging deployment
- No manual steps involved after push → shows practical CI/CD in action
- Demonstrates how teams separate staging from production using environments
